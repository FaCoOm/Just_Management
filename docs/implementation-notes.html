Design Decisions

Manual reservation tab remains a scaffold

2026-05-26T16:08:41.5748040+10:00 · files: src/components/reservations/reservations-page.tsx

Ambiguity: The user asked for manual reservation entry, but the current backend exposes reservation read endpoints and CSV ingestion, not a manual POST /api/reservations creation route.

Choice: Build the manual reservation form as the default modal surface, but keep its submit action disabled with explicit copy explaining that backend support is still required.

Why: This gives operators the intended UX shape without pretending manual writes exist or creating unverified data paths.

CSV upload uses existing reservation ingestion contract

2026-05-26T16:08:41.5748040+10:00 · files: src/components/reservations/reservations-page.tsx, backend/src/ingest/routes.ts

Ambiguity: The requested CSV upload mode did not specify a new endpoint or whether it should reuse the admin integrations surface.

Choice: Reuse POST /api/ingest/reservations with multipart file, sourceAccount, and mandatory dryRun.

Why: That is the current real reservation sync mechanism and already routes through parser, normalizer, reconciliation, dead letters, and sync run tracking.

Document current reservation sync as-is

2026-05-26T16:08:41.5748040+10:00 · files: docs/analysis/reservation-sync-overview.md

Ambiguity: The user asked what the current reservation database sync process is, not necessarily for new backend sync semantics.

Choice: Write an overview of the existing CSV/provider sync path and explicitly call out that manual creation is not yet implemented server-side.

Why: This separates verified current behavior from future implementation requirements.
