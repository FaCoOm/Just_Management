Design Decisions

Target kind comes from inbox subfolder

2026-05-28T03:55:00Z · files: backend/src/ingest/watchers/folder.ts, backend/src/ingest/routes.ts

Ambiguity: Design did not specify whether filename prefixes should remain as fallback routing.

Choice: Removed filename-prefix routing from watcher and route execution. Files under listings/inbox become listings; files under reservations/inbox become reservations.

Why: User chose one root with target subfolders. Keeping filename routing would preserve old ambiguity.

Shared Prisma singleton

2026-05-28T03:50:00Z · files: backend/src/lib/prisma.ts

Ambiguity: Existing index.ts exported a Prisma instance, but lower-level ingest modules should not import Express bootstrap.

Choice: Created backend/src/lib/prisma.ts and imported it from routes, services, watcher, and server startup.

Why: Avoids multiple connection pools and lifecycle coupling.

Deviations

Invalid reservation dates use epoch in trace rows

2026-05-28T04:00:00Z · files: backend/src/ingest/services/reservations.ts

Spec says: Write one provider_reservation_import_rows row per source row before attempting resolution.

Implementation does: Because trace table date columns are non-null, malformed date rows use new Date(0) in trace rows and then mark resolution_status=unresolved.

Reason: Preserves traceability without changing schema nullability in this slice.

Reversibility: Moderate; can make date fields nullable later and remove epoch fallback.

Tradeoffs

Move files during pipeline run, not watcher event

2026-05-28T03:58:00Z · files: backend/src/ingest/routes.ts, backend/src/ingest/watchers/folder.ts
  • A (chosen): watcher only fingerprints files; pipeline run moves to processed/quarantine after processing.
  • B: watcher moves files immediately; simpler inbox cleanup but risks hiding unprocessed files.

Decision driver: Existing ingest rules say watcher must not mutate business tables and pipeline/run is explicit execution boundary.

Open Questions

Trace rows for malformed reservation dates

2026-05-28T04:00:00Z · status: OPEN

Question: Should provider_reservation_import_rows.check_in_date and check_out_date become nullable so malformed rows can be traced without epoch placeholders?

Current assumption: Keep schema additive and use 1970-01-01 only for unresolved malformed rows.

Impact if wrong: Add migration to make those date fields nullable and update trace insert logic.
